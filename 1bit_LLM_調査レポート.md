# 1-bit LLM 調査レポート

> 作成日: 2026-04-17

---

## 1. 1-bit LLM とは

### ひとことで言うと

ChatGPTやClaudeのようなAIチャットモデル（LLM）を、**スマホや安いPCでも動くレベルまで超軽量化した技術**。

### もう少し詳しく

通常のLLMは、1つ1つの「重み」（AIの判断基準となる数値）を **16ビット（65,536段階）** の精度で保存している。これに対して1-bit LLMは、重みを **たった3段階（-1, 0, +1）** だけで表現する。

```
通常のLLM:  重み = 0.37294862...  （16ビット = 65,536段階で表現）
1-bit LLM:  重み = -1, 0, +1      （約1.58ビット = 3段階だけで表現）
```

たった3段階でまともに動くのか？と思うかもしれないが、**最初から3段階で動くように学習する**ことで、驚くほど高い性能を維持できることが分かっている。

### 代表的な1-bit LLMモデル

| モデル | 開発元 | パラメータ数 | 公開時期 |
|--------|--------|-------------|----------|
| **Bonsai-8B** | PrismML（Caltech発） | 82億 | 2026年4月 |
| BitNet b1.58 2B | Microsoft Research | 20億 | 2025年4月 |

本レポートでは、日本語性能が高く商用利用も可能な **Bonsai-8B** を中心に解説する。

### 「1-bitで学習」と「後から1-bitに圧縮」の違い

ここが最も重要なポイント。同じ「1-bit」でもまったく別物の2つの方法がある。

| | 後から圧縮（事後量子化） | 最初から1-bitで学習（1-bit LLM） |
|---|---|---|
| **やること** | 普通に作ったモデルを後から小さくする | **最初から小さい前提で作る** |
| **たとえ** | 高画質写真をJPEGで圧縮 → ぼやける | **最初からドット絵として描く → くっきり** |
| **性能** | 圧縮するほど劣化する | 劣化が少ない |
| **手軽さ** | 既存モデルを変換できる | ゼロから学習し直す必要がある |

**1-bit LLMの核心は「後から圧縮」ではなく「最初から1-bitで学習する」こと。** これにより、極端にデータを小さくしても性能を保てる。

## 2. 1-bit LLM のメリット

### 2.1 とにかく軽い ― メモリ使用量が桁違い

通常の8Bモデル（LLaMA 3など）は約16GBのメモリが必要。Bonsai-8Bはたった **1.15GB** で動く。

| モデル | パラメータ数 | メモリ使用量 | 倍率 |
|--------|-------------|-------------|------|
| LLaMA 3 8B（通常） | 80億 | ~16 GB | 基準 |
| Gemma 4 E4B（4-bit量子化） | 80億（実効40億） | 5.5〜6 GB | 約3分の1 |
| **Bonsai-8B（1-bit）** | **82億** | **1.15 GB** | **約14分の1** |

### 2.2 高価なGPUがなくても動く

通常のLLMを快適に動かすには、数十万円以上するGPU（グラフィックボード）が必要。1-bit LLMなら **普通のCPUやスマートフォンでも実用的な速度で動作する**。

| 動作環境 | Bonsai-8B の速度 |
|---------|-----------------|
| iPhone 17 Pro Max | 44 トークン/秒 |
| iPhone 17 Pro | 約40 トークン/秒 |
| M4 Pro Mac | 131 トークン/秒 |
| RTX 4090（GPU） | 368 トークン/秒 |

> 参考: 人間がテキストを読む速度は約5〜10トークン/秒。スマホでも人間の読む速度の4倍以上で回答を生成できる。

**ハードウェア要件の比較:**

| | 通常のLLM（8B規模） | 1-bit LLM（Bonsai-8B） |
|---|---|---|
| **GPU** | ほぼ必須（8〜16GB VRAM） | **不要** |
| **必要なメモリ** | 16GB以上 | **2GB程度で十分** |
| **動くデバイス** | GPU搭載PC（15万円〜） | **スマホ・一般PC（数万円〜）** |

### 2.3 電気代が大幅に安い

1-bit LLMは計算が単純（足し算・引き算だけ）なので、消費電力が非常に少ない。

| モデル | エネルギー効率 |
|--------|--------------|
| Bonsai-8B | 通常の8Bモデルの **4〜5倍省エネ** |
| BitNet b1.58 2B | 同規模モデルの **約12倍省エネ** |

バッテリーで動くスマホやIoT機器にとって、この差は非常に大きい。

### 2.4 計算が単純 ＝ 速い理由

通常のLLMは「小数点付きの掛け算」を大量に行う。1-bit LLMは重みが -1, 0, +1 だけなので、**掛け算が不要になり、足し算と引き算だけで済む**。

```
通常: 0.3729 × 0.8412 = ?   ← 複雑な掛け算が必要
1-bit:  +1  × 0.8412 = 0.8412  ← そのまま（掛け算不要）
         0  × 0.8412 = 0       ← ゼロ（計算不要）
        -1  × 0.8412 = -0.8412 ← 符号を反転するだけ
```

だからGPUがなくても速く動く。

## 3. 1-bit LLM のデメリット

### 3.1 モデルを作るのが大変

- 既存のモデルを変換するのではなく、**ゼロから学習し直す必要がある**
- この学習には大量のGPUと計算資源が必要（利用する側ではなく、開発する側の問題）
- そのため、新しい1-bit モデルを作れるのは資金力のある研究機関や企業に限られる

### 3.2 まだ大規模モデルがない

- 2026年4月時点で最大の1-bit LLMは **約80億パラメータ**（Bonsai-8B）
- Claude Opus 4.6やGPT-5のような超大規模モデル（推定数百億〜数千億パラメータ）の1-bit版はまだ存在しない
- パラメータが増えたときに1-bit方式でどこまで性能が出るかは未知数

### 3.3 使える道具（ツール）がまだ少ない

- 効率よく動かすには**専用のソフトウェアが必要**（標準ツールでは1-bitの恩恵を受けられない）
- モデルを自分好みに追加学習（ファインチューニング）するノウハウもまだ発展途上
- 対応するサービスやプラットフォームが限られている

### 3.4 性能の天井がある

- 同じ80億パラメータの通常モデルと比べれば互角以上だが、**数百億パラメータのクラウドモデルにはかなわない**
- 高度な推論、長文の処理、プログラミングなど複雑なタスクでは差が出やすい
- あくまで「このサイズにしては驚異的」という評価

## 4. 性能評価 ― 3つのタイプのモデルを比べてみる

「1-bit LLMはどのくらい使えるのか？」を分かりやすくするため、タイプの違う3つのモデルを比較する。

### モデル一覧

| | Bonsai-8B | Gemma 4 E4B | Claude Opus 4.6 |
|---|---|---|---|
| **タイプ** | 1-bit LLM | オンデバイスLLM | クラウドLLM |
| **開発元** | PrismML（Caltech発） | Google DeepMind | Anthropic |
| **パラメータ数** | 82億 | 80億（実効40億） | 非公開（推定 数百億以上） |
| **重みの精度** | 1-bit（2段階） | 16-bit（量子化可） | 16-bit（推定） |
| **メモリ使用量** | 1.15 GB | 5.5〜6 GB | クラウドで処理 |
| **コンテキスト長** | 65,000 トークン | 128,000 トークン | 1,000,000 トークン |
| **動作環境** | スマホ〜PC（GPU不要） | スマホ〜PC | クラウドAPI |
| **利用料金** | 無料（ローカル実行） | 無料（ローカル実行） | API従量課金 |
| **商用利用** | 可（Apache 2.0） | 可 | 可（API経由） |
| **マルチモーダル** | テキストのみ | テキスト+画像+音声+動画 | テキスト+画像 |

### 4.1 賢さの比較（ベンチマークスコア）

ベンチマークとは、AIの能力を測る共通テストのようなもの。

| テスト内容 | Bonsai-8B | Gemma 4 E4B | Claude Opus 4.6 |
|-----------|-----------|-------------|-----------------|
| **知識テスト**（MMLU） | 65.7 | 69.4 | -(※) |
| **専門知識**（GPQA Diamond） | - | 58.6 | 88.5 |
| **数学コンペ**（AIME 2026） | - | 42.5 | - |
| **数学**（MATH） | - | - | 94.2 |
| **コード生成**（HumanEval） | - | - | 97.8 |
| **コード生成**（LiveCodeBench v6） | - | 52.0 | - |
| **総合平均** | 70.5 | - | 92 (※) |

> ※ Claude Opus 4.6はBenchLM.aiの総合スコア92/100。テストの種類が異なるため直接比較は参考値。
> ベンチマークのバージョンや条件が異なるため、厳密な横並び比較ではなく傾向として参照のこと。

**読み方のポイント:**
- Bonsai-8Bの総合スコア70.5は、**同じサイズ（8B）の通常モデルと同等以上**（LLaMA 3 8Bの67.1を上回る）
- ただし、Claude Opus 4.6のようなクラウドの最高性能モデルとは明確な差がある
- Gemma 4 E4Bはマルチモーダル（画像なども扱える）という強みがある

### 4.2 効率の比較

| 比較項目 | Bonsai-8B | Gemma 4 E4B | Claude Opus 4.6 |
|---------|-----------|-------------|-----------------|
| **必要メモリ** | **1.15 GB** | 5.5〜6 GB | クラウド処理 |
| **スマホでの速度** | 44 tok/s | ~20 tok/s | -(※) |
| **Mac（M4 Pro）** | 131 tok/s | - | -(※) |
| **GPU（RTX 4090）** | 368 tok/s | - | -(※) |
| **オフライン利用** | 可能 | 可能 | 不可 |
| **月額コスト（個人利用）** | 0円 | 0円 | 数千〜数万円 |

> ※ クラウドモデルは応答速度が回線環境やサーバー負荷に依存するため単純比較が難しい

### 4.3 3モデルの立ち位置

```
賢さ
  ▲
  │  ★ Claude Opus 4.6
  │     とにかく賢い。ただしネット必須＆有料。
  │
  │     ★ Gemma 4 E4B
  │        かなり賢い。スマホでも動く。画像も扱える。6GB必要。
  │
  │        ★ Bonsai-8B
  │           十分賢い。たった1GBでスマホでも高速。テキスト専用。
  │
  └──────────────────────────────────────────▶ 軽さ・手軽さ
```

### 4.4 まとめ: どう使い分ける？

| 用途 | 最適なモデル | 理由 |
|------|------------|------|
| 最高精度が必要な業務 | **Claude Opus 4.6** | 性能が圧倒的。コスト許容できるなら最善 |
| スマホで画像も扱いたい | **Gemma 4 E4B** | マルチモーダル対応。6GBあれば動く |
| とにかく軽くオフラインで使いたい | **Bonsai-8B** | 1.15GBで十分実用的な性能。無料 |
| IoT・組み込み・大量デプロイ | **Bonsai-8B** | 超省メモリ・省エネが最大の武器 |
| プライバシー重視（データ外部送信NG） | **Bonsai-8B / Gemma 4 E4B** | ローカル実行でデータが外に出ない |

**1-bit LLMの本質的な価値:**

「性能で最強を目指す技術」ではなく、**「今までAIを使えなかった場所にAIを届ける技術」**。高価なGPUがなくても、ネット環境がなくても、スマホ1台で実用的なAIが動く世界を実現する。
